# components.md

## 目的

本文档定义小程序 v0.2 阶段可复用的全局组件规则。

页面专属结构、页面组件组合方式、页面状态和页面验收标准不写在本文档中，应以 `docs/design/page/pages/` 为准。本文档只约束按钮、卡片、标签、表单、空状态、列表项等跨页面复用的基础组件。

## 使用边界

- 全局视觉 token 见 `docs/design/visual-system.md`。
- 页面布局契约见 `docs/design/page/01-page-layout-contract.md`。
- 页面级组件组合见 `docs/design/page/pages/`。
- 数据字段来源见架构文档；本文档只描述组件展示所需的字段类型，不定义数据模型。

## Button

### PrimaryButton

用于页面最主要的提交或推进动作。

样式：

- 高度：44px 起。
- 圆角：`md`。
- 背景：`accent`。
- 文字：`#FFFFFF`。
- 字重：600。
- 点击态：`accent-pressed`。
- 禁用态：`surface-muted` 背景 + `muted` 文字。

规则：

- 单个页面首屏最多出现一个主按钮。
- 文案必须是明确动作，例如「创建目标」「生成计划」「开始今日任务」「保存复盘」。
- 不使用「确定」「提交」「继续」这类脱离上下文的泛化文案。
- 如果按钮禁用，必须通过上下文或错误提示解释原因。

### SecondaryButton

用于次级动作，例如查看、调整、稍后处理。

样式：

- 高度：40px 起。
- 圆角：`md`。
- 背景：`surface`。
- 文字：`ink` 或 `accent`。
- 边框：`1px solid border`。

规则：

- 不抢占主按钮视觉权重。
- 不用于页面最关键的提交动作。
- 同一区域内最多保留两个次级按钮。

### TextButton

用于低权重操作，例如修改、跳过、查看说明。

样式：

- 背景：透明。
- 文字：`accent` 或 `muted`。
- 不加边框。

规则：

- 必须有足够点击区域，不能只让很短文字成为唯一点击点。
- 删除、放弃类操作必须有明确二次确认或低风险回退路径。

### IconButton

用于工具型操作，例如编辑、关闭、展开。

规则：

- 图标必须有可理解的文本上下文或无障碍说明。
- 点击区域不小于 44px。
- 不把关键主操作隐藏成单独小图标。

## Card

### BaseCard

用于承载页面中的主要信息块。

样式：

- 背景：`surface`。
- 边框：`1px solid border`。
- 圆角：`lg`。
- 内边距：`lg`。
- 阴影：默认不使用强阴影。

规则：

- 卡片用于信息分组，不用于制造装饰层级。
- 不要把卡片嵌套在卡片里。
- 单张卡片只承载一个主要意图。

### HighlightCard

用于强调当前最重要的信息，例如今日重点任务、当前目标进度。

样式：

- 背景：优先使用 `surface`，必要时使用 `accent-soft` 或 `warm-soft` 的小面积区域。
- 圆角：`lg` 或 `xl`。
- 可使用左侧 3px `accent` 色条强调。

规则：

- 单个页面首屏最多一个 HighlightCard。
- 强调区域不能大面积铺满高饱和颜色。
- 今日关键词、塔罗、MBTI 只能作为轻量表达元素，不能成为主视觉背景。

### ListCard

用于任务、计划、目标、偏好项等列表内容。

规则：

- 每项必须有清晰主文本。
- 辅助信息最多 2 行，超出应折叠或进入详情。
- 列表项之间保持稳定间距，不用密集分割线制造压迫感。

## Tag

### StatusTag

用于展示任务、计划、目标或能量状态。

样式：

- 圆角：`pill`。
- 字号：`caption`。
- 内边距：水平 8px，垂直 4px。

状态建议：

| 状态 | 推荐视觉 |
|---|---|
| 未开始 | `surface-soft` + `neutral` |
| 进行中 | `accent-soft` + `accent` |
| 已完成 | `success-soft` + `success` |
| 部分完成 | `warning-soft` + `warning` |
| 已跳过 | `surface-soft` + `muted` |
| 严重错误 | `danger-soft` + `danger` |

规则：

- 颜色不能作为唯一状态信号，必须搭配文字。
- 普通未完成任务不使用 `danger`。
- `skipped` 不使用惩罚性文案或红色视觉。

### OptionChip

用于轻量选择，例如能量状态、可用时间、计划强度、偏好时段。

样式：

- 未选中：`surface` + `border`。
- 选中：`accent-soft` + `accent`。
- 圆角：`pill` 或 `md`。

规则：

- 选项必须短，避免长句塞入 chip。
- 多选和单选必须在视觉上区分清楚。
- 不把复杂解释放入 chip 内部。

## FormField

用于目标创建、偏好设置、复盘备注等输入区域。

必须包含：

- Label。
- 输入控件。
- 必要时的 helper text 或 error text。

规则：

- 必填项要清楚，但不要制造压迫感。
- 错误提示必须具体说明如何修正。
- helper text 控制在一句话内。
- 长文本输入默认不占据首屏过多空间。

## EmptyState

用于无目标、无计划、无今日任务等场景。

必须包含：

- 温和标题。
- 一句具体说明。
- 一个下一步动作。

规则：

- 不使用「暂无数据」作为唯一内容。
- 不使用责备性文案。
- 空状态必须引导用户做下一步，例如「创建目标」或「查看计划」。
- 空状态插图不是 MVP 必须；如使用插图，应保持轻量，不抢占页面主体。

## FeedbackNotice

用于成功、错误、加载、计划不可行等反馈。

规则：

- 成功反馈短而明确。
- 错误反馈必须包含问题和下一步。
- AI 失败时应提示已使用基础规则或允许重试，不直接中断流程。
- 计划不可行时应给出可选动作：延长 DDL、增加每日可用时间、缩小目标范围。

## TaskCard Shell

任务卡片是跨页面复用的信息外壳，具体内容权重由页面文档决定。

推荐字段：

- 任务标题。
- 预计用时。
- 最低完成线。
- 当前状态。
- 主要动作。

规则：

- 最低完成线必须具体、低压力、可执行。
- 每张任务卡只放一个主要动作。
- `estimatedMinutes` 可见。
- 今日页是否突出一个主任务，以 `docs/design/page/pages/today-page.md` 为准。
- 日历页任务是否按日期聚合，以 `docs/design/page/pages/calendar-page.md` 为准。

## PreferenceGroup

用于展示和编辑默认安排偏好、AI 表达偏好、仪式感元素。

规则：

- 安排偏好和表达偏好必须分开。
- MBTI、塔罗、每日关键词只属于表达偏好或仪式感偏好。
- 不把 MBTI、塔罗、每日关键词作为任务排序、任务难度或计划可行性的硬参数。

## 禁止项

- 不要在组件文档里定义某个页面完整结构。
- 不要让卡片嵌套卡片。
- 不要让一个组件同时承载多个主要动作。
- 不要用大段 AI 解读占据首屏。
- 不要用红色表达普通未完成、跳过或低能量。
- 不要把页面导航入口伪装成普通内容卡片重复出现。
