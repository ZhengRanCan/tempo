# 个性化 AI 任务日历小程序：目标、计划、任务状态设计报告

## 1. 报告背景

当前小程序的产品定位是「个性化 AI 任务日历」。用户不是单纯创建一个待办事项，而是输入一个长期目标、截止日期、每日可用时间和状态偏好后，由系统自动将大目标拆解成每天可执行的小任务，并生成每日任务日历。

系统后续还需要根据用户昨日任务完成情况、当天状态和晚间复盘结果，动态调整后续任务安排。因此，本小程序不能只设计「任务状态」，还需要明确区分：

1. 目标 Goal
2. 计划 Plan
3. 阶段 Stage
4. 任务 Task

如果不区分这些对象，后续会出现以下问题：

- 用户完成了一个今日任务，但系统误以为整个目标完成了。
- 用户取消了某个任务，但系统误以为整个目标被取消了。
- AI 重排任务时，无法说明是重排了计划，还是修改了目标。
- 今日任务页、任务日历页、复盘页对“状态”的理解不一致。
- 长期目标无法支持「近 7 天具体任务 + 远期阶段安排」的计划模式。

因此，本报告建议将系统状态设计拆成三层：

- Goal 管最终目标。
- Plan 管时间安排。
- Task 管具体执行。

---

## 2. 核心结论

本小程序中，用户创建的是「目标」，不是「任务」。

AI 接收目标后，应该先生成「计划」。计划中包含两类内容：

1. 未来 7 天的具体任务；
2. 截止日期之前的粗略阶段安排。

用户每天真正操作的是「任务」，例如开始专注、标记完成、今天做不了、取消任务等。

目标是否完成，不取决于某一个任务是否完成，而取决于目标下的核心阶段、关键任务是否完成，或者用户是否手动确认目标已经达成。

推荐采用以下结构：

```text
Goal 目标
  └── Plan 计划
        ├── Stage 阶段安排
        └── Task 每日任务
```

简单理解：

- 目标是目的地。
- 计划是路线图。
- 任务是今天要走的一小步。

---

## 3. 对象分层设计

### 3.1 Goal：目标

Goal 表示用户最终要完成的内容。

示例：

- 完成开题报告初稿
- 准备英语考试
- 完成论文实验
- 一个月内完成课程项目

Goal 不应该直接等同于某一天的任务。它是一个较大的、需要被拆解和排期的对象。

Goal 应该包含：

- 目标名称
- 截止日期
- 目标说明
- 每日可用时间
- 用户偏好
- 当前目标状态
- 当前计划 ID
- 目标完成进度

推荐字段：

```ts
type Goal = {
  id: string
  title: string
  description?: string
  deadline: string
  dailyAvailableMinutes?: number
  preferredTime?: 'morning' | 'afternoon' | 'evening' | 'any'
  status: GoalStatus
  activePlanId?: string
  progressPercent?: number
  createdAt: string
  completedAt?: string
}
```

---

### 3.2 Plan：计划

Plan 表示 AI 针对某个 Goal 生成的时间安排。

一个 Goal 可以有多个 Plan。比如：

- Plan v1：用户刚创建目标时生成的初始计划。
- Plan v2：用户连续两天未完成任务后，AI 重新生成的计划。
- Plan v3：用户修改了 DDL 或每日可用时间后，重新生成的计划。

Plan 的作用是保存「当前目标如何被安排到时间轴上」。

推荐字段：

```ts
type Plan = {
  id: string
  goalId: string
  version: number
  status: PlanStatus
  startDate: string
  endDate: string
  totalTaskCount: number
  completedTaskCount: number
  totalEstimatedMinutes: number
  remainingEstimatedMinutes: number
  progressPercent: number
  createdAt: string
  updatedAt: string
}
```

---

### 3.3 Stage：阶段

Stage 用于长期目标的粗粒度安排。

如果用户设置的 DDL 在一个月后，不建议 AI 一次性把未来 30 天全部拆成固定任务。更合理的方式是：

- 未来 7 天：生成具体任务。
- 7 天以后：生成阶段安排。
- 每天复盘后：滚动细化后续任务。

示例：

- 第 1 周：资料整理与研究问题确认
- 第 2 周：完成初稿主体
- 第 3 周：补充案例和实验设计
- 第 4 周：修改、格式整理、预留缓冲

推荐字段：

```ts
type Stage = {
  id: string
  planId: string
  goalId: string
  title: string
  startDate: string
  endDate: string
  description?: string
  status: StageStatus
}
```

---

### 3.4 Task：任务

Task 是用户每天真正执行的小步骤。

示例：

- 整理 3 篇核心文献摘要
- 补充 2 篇参考文献
- 写研究背景第一段
- 梳理研究问题

Task 应该是可执行的，最好能在 15-60 分钟内完成。如果任务太大，复盘时应该允许 AI 将其拆小。

推荐字段：

```ts
type Task = {
  id: string
  goalId: string
  planId: string
  stageId?: string

  title: string
  minDoneLine: string
  scheduledDate: string
  originalDate?: string

  estimatedMinutes: number
  priority: 'high' | 'medium' | 'low'
  type: 'focus' | 'normal' | 'backup'

  status: TaskStatus

  rescheduleCount: number
  rescheduleReason?: string
  missedReason?: string
  completedAt?: string

  createdAt: string
  updatedAt: string
}
```

## 4. 目标状态设计

GoalStatus 用于描述整个目标处在什么阶段。

推荐状态：

```ts
type GoalStatus =
  | 'draft'        // 草稿，用户还没提交完整目标信息
  | 'planning'     // AI 正在生成计划
  | 'active'       // 目标进行中
  | 'at_risk'      // 目标有延期风险
  | 'completed'    // 目标已完成
  | 'paused'       // 用户暂停目标
  | 'cancelled'    // 用户取消整个目标
```

### 4.1 状态说明

#### draft

用户正在填写目标，还没有提交。

#### planning

目标已提交，AI 正在生成初始计划。

#### active

目标已经生成计划，正在执行。

#### at_risk

目标存在延期风险，例如剩余任务过多、连续多天未完成、剩余时间不足。

#### completed

目标完成。可以由用户手动确认，也可以在关键任务全部完成后由系统建议确认。

#### paused

用户暂时暂停目标，但未来可能继续。

#### cancelled

用户明确取消整个目标。

### 4.2 目标状态边界

注意：

- 取消任务不等于取消目标。
- 完成任务不等于完成目标。
- 目标状态应该服务于整体目标进度，而不是单个任务执行。

---

## 5. 计划状态设计

PlanStatus 用于描述 AI 生成的计划是否有效、是否需要重排。

推荐状态：

```ts
type PlanStatus =
  | 'generating'     // 正在生成计划
  | 'active'         // 当前计划生效中
  | 'needs_replan'   // 需要重排
  | 'replanning'     // 正在重排
  | 'completed'      // 计划已完成
  | 'obsolete'       // 旧计划已废弃
```

### 5.1 状态说明

#### generating

AI 正在生成计划。

#### active

计划当前有效，今日任务和日历页都基于该计划展示。

#### needs_replan

由于任务未完成、状态变化、DDL 修改等原因，计划需要重新安排。

#### replanning

AI 正在根据新情况重排计划。

#### completed

计划内任务已经完成，或目标已经完成。

#### obsolete

旧计划被新计划替代，不再作为当前计划使用。

### 5.2 计划状态边界

注意：

- 一个目标可以有多个计划版本，但同一时间只应该有一个 active plan。
- 重排计划时，不建议直接覆盖所有历史信息，至少要保留 plan version 或重排历史。
- 计划重排不等于目标失败，它是产品闭环中的正常机制。

---

## 6. 任务状态设计

TaskStatus 用于描述单个任务的执行情况。

推荐 MVP 状态：

```ts
type TaskStatus =
  | 'not_started'   // 未开始
  | 'in_progress'   // 进行中
  | 'completed'     // 已完成
  | 'missed'        // 当天未完成
  | 'cancelled'     // 已取消
```

### 6.1 状态说明

#### not_started

任务已安排到某一天，但用户尚未开始。

#### in_progress

用户点击了「开始专注」，任务正在推进。

#### completed

用户点击「标记完成」，或复盘时确认完成。

#### missed

当天没有完成，等待复盘或重排。

#### cancelled

用户确认这个任务不再需要执行。

### 6.2 是否需要 rescheduled 状态

MVP 阶段可以暂时不把 `rescheduled` 作为任务主状态。

更推荐的做法是：

- 任务被重排后，更新 `scheduledDate`；
- 增加 `rescheduleCount`；
- 记录 `rescheduleReason`；
- 然后任务状态重新回到 `not_started`。

例如：

```ts
{
  title: '整理 3 篇核心文献摘要',
  originalDate: '2026-05-24',
  scheduledDate: '2026-05-25',
  status: 'not_started',
  rescheduleCount: 1,
  rescheduleReason: '用户 05/24 未完成，顺延到次日'
}
```

这样可以避免任务列表中出现大量“已重排任务”，也更适合 MVP。

### 6.3 逾期状态处理

逾期不建议作为永久数据库状态。

逾期更适合作为根据日期动态计算出来的显示状态。

例如：

```ts
if (task.scheduledDate < today && task.status !== 'completed') {
  displayStatus = 'overdue'
}
```

也就是说：

- 数据状态：`not_started` / `missed`
- UI 显示状态：逾期 / 待处理

## 7. 完整业务流程

### 7.1 用户创建目标

用户填写：

- 目标名称：完成开题报告初稿
- 截止日期：一个月后
- 目标说明：需要完成背景、意义、方法、参考文献
- 每日可用时间：30-60 分钟
- 偏好：晚上更适合工作

系统生成：

```ts
Goal.status = 'planning'
Plan.status = 'generating'
```

---

### 7.2 AI 生成计划

AI 生成两层内容。

第一层：未来 7 天具体任务。

例如：

- 今天：整理 3 篇核心文献摘要
- 明天：整理研究背景材料
- 后天：补充参考文献
- 第 4 天：写研究意义草稿

第二层：一个月粗略阶段安排。

例如：

- 第 1 周：资料整理与研究问题确认
- 第 2 周：完成初稿主体
- 第 3 周：补充案例和实验设计
- 第 4 周：修改、格式整理、预留缓冲

生成完成后：

```ts
Goal.status = 'active'
Plan.status = 'active'
Task.status = 'not_started'
```

---

### 7.3 用户查看今日任务

今日任务页读取：

```ts
Task.scheduledDate === today
```

如果今天有多个任务，系统从中选择一个「今日重点任务」。

今日重点任务可以依据：

- 优先级高
- DDL 更近
- 是否为关键路径任务
- 是否适合用户当前状态
- 是否属于昨天顺延任务

今日页展示的是 Task，不是整个 Goal。

---

### 7.4 用户执行任务

用户点击「开始专注」：

```ts
Task.status = 'in_progress'
```

用户点击「标记完成」：

```ts
Task.status = 'completed'
Task.completedAt = now
```

系统同步更新：

```ts
Plan.completedTaskCount += 1
Plan.progressPercent 更新
Goal.progressPercent 更新
```

---

### 7.5 用户今天做不了

用户点击「今天做不了」，或晚间复盘中选择未完成：

```ts
Task.status = 'missed'
Task.missedReason = '时间不够' | '状态不好' | '任务太难' | '临时有事'
Plan.status = 'needs_replan'
```

AI 根据原因决定：

- 顺延任务
- 拆小任务
- 降低最低完成线
- 调整后续日期
- 取消不必要任务

---

### 7.6 AI 重排任务

如果只是顺延任务：

```ts
Task.scheduledDate = newDate
Task.status = 'not_started'
Task.rescheduleCount += 1
Task.rescheduleReason = '用户未完成，顺延到下一可用日期'
Plan.status = 'active'
```

如果任务太大，需要拆分，原任务可以处理为：

```ts
Task.status = 'cancelled'
Task.cancelReason = '任务过大，已拆分为多个小任务'
```

然后生成新的小任务：

```text
Task A：整理文献标题
Task B：写出 3 条摘要要点
Task C：整合成一段背景材料
```

---

### 7.7 目标完成

目标完成不应该只依赖某一个任务完成，而应根据以下条件判断：

- 关键任务全部完成；
- 核心阶段全部完成；
- 用户手动确认目标完成；
- AI 判断目标已达到最低完成标准，并请求用户确认。

完成后：

```ts
Goal.status = 'completed'
Plan.status = 'completed'
Goal.completedAt = now
```

---

## 8. 页面职责映射

### 8.1 创建目标页

负责 Goal。

页面重点：

- 输入目标名称
- 输入截止日期
- 输入目标说明
- 设置每日可用时间
- 设置状态偏好
- 提交后触发 AI 生成计划

不应该在这里直接操作 Task 状态。

---

### 8.2 任务日历页

负责 Plan。

页面重点：

- 当前目标计划卡
- 整体进度
- 时间压力
- 未来 7 天详细任务
- 长期阶段安排
- 某一天任务列表
- 调整计划 / 重新生成计划入口

日历页展示的是「目标如何被排到时间上」。

---

### 8.3 今日任务页

负责 Task。

页面重点：

- 今日重点任务
- 今日全部任务入口
- 当前状态调整
- AI 今日建议
- 开始专注
- 标记完成
- 今天做不了

今日页展示的是「今天具体做什么」。

---

### 8.4 晚间复盘页

负责 Task + Plan。

页面重点：

- 今天完成了哪些任务
- 哪些任务没完成
- 未完成原因
- 是否需要重排
- AI 生成明天或后续调整建议

复盘页是任务状态进入重排逻辑的关键入口。

---

### 8.5 我的页

负责用户偏好和目标概览。

页面重点：

- 当前目标
- 常用可用时间
- 能量状态偏好
- 工作时段偏好
- MBTI / 塔罗 / 每日关键词等轻量偏好

注意：MBTI、塔罗、每日关键词只能作为表达风格和仪式感包装，不应作为严肃预测或任务决策依据。

## 9. 状态边界说明

### 9.1 任务取消不等于目标取消

用户取消一个任务，可能只是因为该任务不再必要，或者 AI 后续会用更合适的小任务替代它。

只有用户明确取消整个目标时，Goal 才进入 `cancelled`。

---

### 9.2 任务完成不等于目标完成

用户完成「整理 3 篇文献摘要」，只能说明某个 Task 完成。

目标「完成开题报告初稿」仍然可能处于 `active`。

---

### 9.3 计划重排不等于目标失败

任务未完成后触发重排，是产品的正常机制，不应该被设计成失败状态。

用户看到的文案应避免过强挫败感。

推荐：

- 已帮你顺延到明天
- 计划已调整
- 今天先完成最低完成线即可

不推荐：

- 任务失败
- 计划失败
- 目标失败

---

### 9.4 逾期应优先作为显示状态

不要过早把 `overdue` 做成永久数据库状态。

推荐用日期动态计算：

```ts
const isOverdue = task.scheduledDate < today && task.status !== 'completed'
```

UI 上可以显示「逾期 / 待重排」，但数据上仍然可以保留原始任务状态。

## 10. 建议的 MVP 状态模型

MVP 第一版不需要设计得过重，建议先实现以下最小闭环。

### 10.1 GoalStatus

```ts
type GoalStatus =
  | 'planning'
  | 'active'
  | 'at_risk'
  | 'completed'
  | 'paused'
  | 'cancelled'
```

### 10.2 PlanStatus

```ts
type PlanStatus =
  | 'generating'
  | 'active'
  | 'needs_replan'
  | 'replanning'
  | 'completed'
  | 'obsolete'
```

### 10.3 TaskStatus

```ts
type TaskStatus =
  | 'not_started'
  | 'in_progress'
  | 'completed'
  | 'missed'
  | 'cancelled'
```

这套状态已经足够支持：

- 创建目标
- AI 生成计划
- 展示今日任务
- 用户执行任务
- 标记完成
- 记录未完成
- 晚间复盘
- 自动重排
- 日历页展示计划进度

## 11. 对 UI 的影响

### 11.1 今日任务页

今日任务页主要显示 Task 状态。

推荐状态展示：

- 未开始
- 进行中
- 已完成
- 今天做不了
- 已顺延
- 已取消

今日页重点不是展示目标全貌，而是让用户快速执行今日任务。

---

### 11.2 任务日历页

任务日历页主要显示 Plan 状态。

推荐状态展示：

- 当前目标进度
- 计划正常
- 计划偏紧
- 有延期风险
- 某天任务偏多
- 某天是缓冲日
- 某任务已顺延

日历页重点不是执行任务，而是查看整体排布和调整计划。

---

### 11.3 复盘页

复盘页需要记录任务结果和原因。

推荐复盘选项：

- 完成
- 部分完成
- 没开始
- 时间不够
- 状态不好
- 任务太难
- 临时有事
- 已经不需要

这些不一定都变成 TaskStatus，而应作为 `reviewResult` 或 `missedReason` 记录。

---

## 12. 开发实现建议

1. 先统一数据概念，不要把 Goal 和 Task 混用。
2. 目标创建后，先生成 Goal，再生成 Plan，再生成 Task。
3. 今日页只读取当天 Task。
4. 日历页读取当前 Goal 的 active Plan。
5. 长期目标采用「近 7 天细任务 + 远期阶段计划」的方式。
6. 每次复盘后，如果有未完成任务，将 Plan 标记为 `needs_replan`。
7. AI 重排完成后，将新计划设为 active，旧计划设为 obsolete；或者在 MVP 中直接更新当前 plan 并记录 version。
8. 任务重排时优先保留同一个 Task，只更新 scheduledDate 和 rescheduleCount，避免生成大量重复任务。
9. 如果任务过大，可以取消原任务并拆分生成多个新任务。
10. 所有状态文案都应避免制造失败感，强调「调整」和「继续推进」。

---

## 13. 验收标准

完成状态模型设计后，需要满足以下标准：

1. 系统能明确区分 Goal、Plan、Task。
2. 用户完成一个 Task 后，不会误判 Goal 已完成。
3. 用户取消一个 Task 后，不会误判 Goal 已取消。
4. AI 可以基于未完成任务触发 Plan 重排。
