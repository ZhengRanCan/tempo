# 创建目标页 UI 优化报告

## 1. 报告背景

当前小程序已经有顶部导航栏和底部 TabBar，因此「创建目标」页不需要再在页面主体中重复展示大标题、页面说明副标题或胶囊标签。页面主体应该直接进入创建目标流程。

本次优化基于当前讨论中更认可的右侧版式：页面保持清爽，但将「个性化安排偏好」完整展开展示，让用户在创建目标时可以一次性完成关键设置。

本页的核心任务不是做一个复杂表单，而是帮助用户快速完成三件事：

1. 说清楚自己想完成什么；
2. 设定截止日期；
3. 设定每天大概能投入多少时间，并补充必要偏好。

创建目标页应该降低用户的创建成本，让用户感觉“写下目标后，系统会帮我拆解计划”，而不是让用户感觉自己在填写复杂配置表。

---

## 2. 页面定位

「创建目标」页负责创建 Goal，即用户最终想完成的目标。

它不应该负责展示今日任务，也不应该负责展示完整计划详情。创建目标页提交后，系统再进入 AI 生成计划流程，并生成 Plan、Stage 和 Task。

页面定位可以概括为：

> 用户输入目标，系统准备生成计划。

对应产品结构：

```text
创建目标页：创建 Goal
任务日历页：展示 Plan
今日任务页：执行 Task
晚间复盘页：反馈 Task 并触发 Plan 重排
```

因此，创建目标页的 UI 应该围绕 Goal 创建展开，而不是把计划、任务、复盘等功能混在一起。

---

## 3. 右侧版式的优点

本次更推荐右侧版式，原因如下：

### 3.1 信息层级更清楚

右侧版式按步骤组织内容：

1. 你想完成什么？
2. 什么时候完成？
3. 每天大概能投入多久？
4. 补充说明
5. 个性化安排偏好
6. 生成计划

这种结构比原来的“大表单卡片”更容易理解，用户知道自己正在一步步完成目标创建。

### 3.2 必填项和可选项有区分

页面前三块是核心必填项：

- 目标名称
- 截止日期
- 每天可用时间

后面的「补充说明」和「个性化安排偏好」属于增强项。虽然右侧版式展开了个性化偏好，但它仍然通过模块标题和说明文案表达出“这是偏好，不是硬性要求”。

### 3.3 选择项更像按钮

新版中时间、工作时段、能量状态、计划强度等选项使用矩形胶囊按钮，而不是原来的圆形按钮。这样更符合小程序表单交互习惯，也更容易点击。

### 3.4 页面更像创建流程，而不是设置页

右侧版式用编号、卡片和按钮引导用户完成创建过程。它比原来“字段堆叠 + 偏好设置混杂”的结构更轻。

### 3.5 个性化偏好可以完整展示

用户已经明确偏好右侧版式，因此本轮不采用默认折叠偏好卡的方案，而是保留右侧版式中的展开状态，让用户直接看到：

- 偏好工作时段
- 当前能量状态
- 计划强度
- 仪式感元素

这样有利于突出产品的“个性化 AI 任务安排”特色。

---

## 4. 页面整体结构

优化后的「创建目标」页建议采用以下结构：

```text
顶部导航栏：创建目标

页面主体：
  引导区
  目标名称卡片
  截止日期卡片
  每日可用时间卡片
  补充说明卡片
  个性化安排偏好卡片
  生成计划按钮
  底部安全提示

底部 TabBar：
  今日 / 日历 / 创建 / 我的
```

注意：

- 顶部导航栏已经显示「创建目标」。
- 页面主体不再重复显示大号「创建目标」标题。
- 页面主体可以保留一句轻量引导文案，但不要像原来那样出现长副标题。
- 页面内容应尽量首屏可理解，整体纵向滚动不要过重。

---

## 5. 详细 UI 设计说明

### 5.1 顶部导航栏

保留现有顶部导航栏。

要求：

- 中间标题：创建目标
- 不要在页面主体重复出现「创建目标」大标题
- 不要保留原来的页面说明副标题
- 不要保留「Tempo」或其他胶囊标签

顶部导航栏只负责说明当前页面身份。

---

### 5.2 页面引导区

右侧版式中有一个轻量引导区，可以保留，但需要控制高度。

推荐文案：

```text
先设定一个目标
我会帮你拆解计划，每天专注推进
```

设计要求：

- 可以搭配一个轻量插画或小图标；
- 不要占用太多高度；
- 不要写成长段说明；
- 不要重复顶部导航栏的「创建目标」。

引导区的作用是降低用户心理门槛，而不是解释完整产品逻辑。

---

### 5.3 目标名称卡片

模块标题：

```text
1  你想完成什么？
```

输入框 placeholder：

```text
例如：完成开题报告初稿
```

辅助说明：

```text
写下一个清晰的目标，越具体越好。
```

设计要求：

- 输入框高度适中，不要过厚；
- 建议增加字数提示，例如 0 / 50；
- 目标名称为必填项；
- 输入后应更新按钮可用状态；
- 不要把目标名称裸露显示在页面顶部，目标名只应出现在有上下文的组件中。

字段建议：

```ts
goal.title: string
```

校验建议：

- 非空；
- 长度 2-50 字；
- 不建议只输入纯数字或无意义字符；
- 为空时不允许生成计划。

---

### 5.4 截止日期卡片

模块标题：

```text
2  什么时候完成？
```

输入框 placeholder：

```text
选择截止日期
```

辅助说明：

```text
建议设置一个有挑战但可实现的时间。
```

设计要求：

- 点击后打开日期选择器；
- DDL 不允许早于今天；
- DDL 当天可以安排任务；
- 选择日期后显示具体日期，例如 2026-06-24；
- 如果截止日期距离今天较远，后续由 AI 生成“近 7 天细任务 + 远期阶段安排”。

字段建议：

```ts
goal.deadline: string
```

校验建议：

- 必填；
- 不早于今天；
- 如果 DDL 非常近，应提示计划可能偏紧；
- 如果 DDL 很远，不要一次性生成所有每日任务，应采用阶段计划。

---

### 5.5 每日可用时间卡片

模块标题：

```text
3  每天大概能投入多久？
```

辅助说明：

```text
按每天可用时间生成更合理的计划。
```

推荐选项：

```text
15 分钟
30 分钟
45 分钟
60 分钟
自定义分钟数
```

设计要求：

- 使用矩形胶囊按钮，不要使用圆形按钮；
- 默认可选 30 分钟；
- 选中态使用紫色边框和浅紫背景；
- 自定义分钟数可以展开输入框或弹窗；
- 不要强迫用户填写非常精确的总工作量。

字段建议：

```ts
goal.dailyAvailableMinutes: number
```

校验建议：

- 必填；
- 默认 30；
- 自定义范围建议 5-180 分钟；
- 输入异常时提示用户修正。

---

### 5.6 补充说明卡片

模块标题：

```text
补充说明（可选）
```

辅助说明：

```text
可写下背景、范围、最想先完成的部分等。
```

输入框 placeholder：

```text
比如：需要完成背景、意义、方法、参考文献等
希望重点先完成文献整理部分
```

设计要求：

- 这是可选项；
- 输入框高度建议 3-4 行；
- 建议增加字数提示，例如 0 / 200；
- 不要做得太高，否则页面会显得沉重；
- 用户不填写也可以生成计划。

字段建议：

```ts
goal.description?: string
```

用途：

- 帮助 AI 理解目标范围；
- 帮助 AI 判断任务拆解顺序；
- 帮助 AI 生成更贴近用户意图的阶段计划。

---

### 5.7 个性化安排偏好卡片

用户更喜欢右侧完整展开版式，因此本轮建议默认展开「个性化安排偏好」。

模块标题：

```text
个性化安排偏好
```

该卡片用于设置 AI 安排任务时的表达风格和轻量偏好。它不应该变成严肃心理测评，也不应该让 MBTI、塔罗等元素决定计划结果。

#### 5.7.1 偏好工作时段

标题：

```text
偏好工作时段
```

选项：

```text
上午
下午
晚上
都可以
```

默认：

```text
都可以
```

字段建议：

```ts
goal.preferredTime: 'morning' | 'afternoon' | 'evening' | 'any'
```

用途：

- 影响任务推荐时段；
- 不强制限制任务日期；
- 如果用户选择“晚上”，今日建议可以更偏向晚间执行。

---

#### 5.7.2 当前能量状态

标题：

```text
当前能量状态
```

选项：

```text
低能量
普通
高能量
```

默认：

```text
普通
```

字段建议：

```ts
goal.initialEnergy: 'low' | 'normal' | 'high'
```

用途：

- 影响初始计划强度；
- 低能量时优先生成更短、更低压力的任务；
- 高能量时可以适当安排更高推进度。

---

#### 5.7.3 计划强度

标题：

```text
计划强度
```

选项：

```text
轻松
均衡
冲刺
```

默认：

```text
均衡
```

字段建议：

```ts
goal.planIntensity: 'easy' | 'balanced' | 'sprint'
```

用途：

- 轻松：保留更多缓冲日；
- 均衡：按正常节奏安排；
- 冲刺：在 DDL 较近时可安排更密集任务。

---

#### 5.7.4 仪式感元素

标题：

```text
仪式感元素
```

选项：

```text
每日关键词
塔罗灵感
都不需要
```

默认建议：

```text
每日关键词
```

字段建议：

```ts
goal.ritualElement: 'daily_keyword' | 'tarot_hint' | 'none'
```

说明文案：

```text
仅作为风格参考，不会影响任务安排的科学性。
```

设计边界：

- 不要把塔罗、MBTI、每日关键词做成严肃预测；
- 不要让塔罗结果决定任务是否应该执行；
- 它们只作为提醒风格、仪式感和文案包装。

---

### 5.8 生成计划按钮

按钮文案：

```text
生成计划
```

设计要求：

- 使用紫色主按钮；
- 宽度接近页面主内容宽度；
- 放在页面底部；
- 可以随页面滚动，也可以做成底部固定按钮；
- 如果固定底部，需要避免遮挡 TabBar；
- 未满足必填项时置灰禁用；
- 满足条件后变为可点击状态。

启用条件：

```text
目标名称已填写
截止日期已选择
每日可用时间已选择或填写
```

点击后逻辑：

```text
1. 创建 Goal
2. 进入 Plan generating 状态
3. 调用 AI 生成计划
4. 生成未来 7 天具体 Task
5. 生成 DDL 前的粗略 Stage
6. 跳转到任务日历页或今日任务页
```

建议点击后显示 loading：

```text
正在为你生成计划...
```

---

### 5.9 底部提示

按钮下方可以保留轻量说明：

```text
生成后可随时调整
```

设计要求：

- 字号小；
- 灰色；
- 不要喧宾夺主；
- 不要写成长说明。

---

## 6. 交互状态设计

### 6.1 初始状态

用户未填写任何内容时：

- 目标名称为空；
- 截止日期为空；
- 每日可用时间默认 30 分钟或未选；
- 偏好工作时段默认「都可以」；
- 当前能量默认「普通」；
- 计划强度默认「均衡」；
- 仪式感元素默认「每日关键词」；
- 生成计划按钮禁用。

---

### 6.2 填写中状态

用户输入目标名称后：

- 字数统计更新；
- 输入框保持正常状态；
- 如果输入超过限制，显示提示。

用户选择截止日期后：

- 显示具体日期；
- 如果 DDL 太近，提示“计划可能偏紧”；
- 如果 DDL 较远，正常保存。

用户选择每日可用时间后：

- 当前选项高亮；
- 自定义时间输入后替换当前值。

---

### 6.3 可提交状态

当以下字段均有效时：

- 目标名称有效；
- 截止日期有效；
- 每日可用时间有效；

按钮变为可点击。

点击「生成计划」后：

- 按钮进入 loading；
- 防止重复点击；
- 创建目标并触发 AI 生成计划。

---

### 6.4 生成失败状态

如果 AI 生成计划失败，应显示轻量错误提示：

```text
计划生成失败，请稍后重试。
```

操作：

```text
重新生成
```

不要清空用户已填写内容。

---

## 7. 视觉规范建议

### 7.1 整体风格

延续当前小程序风格：

- 暖白 / 米色背景；
- 白色圆角卡片；
- 紫色作为主色；
- 浅紫用于选中态；
- 灰色用于辅助说明；
- 卡片之间保持统一间距；
- 页面整体保持轻量，不要有厚重后台表单感。

---

### 7.2 卡片样式

建议：

```text
背景：#FFFFFF
圆角：20rpx - 28rpx
边框：浅米色 / 浅灰色
阴影：轻微或不使用
内边距：24rpx - 32rpx
卡片间距：20rpx - 28rpx
```

---

### 7.3 按钮样式

普通选项按钮：

```text
圆角矩形
高度适中
浅边框
未选中：白色背景 + 深灰文字
选中：浅紫背景 + 紫色边框 + 紫色文字
```

主按钮：

```text
紫色背景
白色文字
大圆角
高度 88rpx 左右
禁用态为浅紫或浅灰
```

---

### 7.4 文案风格

文案要轻，避免长说明。

推荐：

```text
写下一个清晰的目标，越具体越好。
建议设置一个有挑战但可实现的时间。
按每天可用时间生成更合理的计划。
生成后可随时调整。
```

避免：

```text
请完整填写所有字段，否则系统无法基于你的状态和个性化偏好生成科学合理的任务计划。
```

创建目标页不需要大段解释。

---

## 8. 数据字段建议

本页主要创建 Goal，同时为 Plan 生成提供初始参数。

建议提交结构：

```ts
type CreateGoalPayload = {
  title: string
  deadline: string
  dailyAvailableMinutes: number
  description?: string

  preferredTime: 'morning' | 'afternoon' | 'evening' | 'any'
  initialEnergy: 'low' | 'normal' | 'high'
  planIntensity: 'easy' | 'balanced' | 'sprint'
  ritualElement: 'daily_keyword' | 'tarot_hint' | 'none'
}
```

创建成功后建议生成：

```ts
Goal.status = 'planning'
Plan.status = 'generating'
```

AI 生成完成后：

```ts
Goal.status = 'active'
Plan.status = 'active'
```

---

## 9. 页面跳转建议

点击「生成计划」后有两种可选方案。

### 方案 A：跳转到任务日历页

适合强调“AI 已经帮你生成整体计划”。

流程：

```text
创建目标 -> 生成计划 -> 跳转任务日历页
```

优点：

- 用户能马上看到未来 7 天计划和长期阶段安排；
- 更符合“目标生成计划”的心智。

### 方案 B：跳转到今日任务页

适合强调“马上开始执行”。

流程：

```text
创建目标 -> 生成计划 -> 跳转今日任务页
```

优点：

- 用户能马上看到今天最该做什么；
- 更利于快速进入执行。

### 推荐

MVP 阶段更推荐方案 A：

```text
创建目标 -> 生成计划 -> 任务日历页
```

因为用户刚创建一个目标，更自然的问题是“系统怎么帮我安排这个目标”。任务日历页更适合承接这个结果。

任务日历页中再提供「查看今日任务」入口。

---

## 10. 本轮不要做的事情

本轮只优化「创建目标」页 UI 和交互，不做以下事情：

1. 不重构整个小程序。
2. 不修改底部 TabBar 架构。
3. 不重新设计今日任务页、任务日历页、我的页。
4. 不把创建目标页做成复杂设置页。
5. 不要求用户填写完整心理测评或人格测试。
6. 不让塔罗、MBTI、每日关键词决定任务安排。
7. 不一次性要求用户填写目标总工作量。
8. 不把未来一个月每天任务全部在创建页展示。
9. 不破坏已有创建目标、生成计划、任务完成、任务重排逻辑。

---

## 11. 验收标准

完成后请确保：

1. 页面顶部只保留顶部导航栏中的「创建目标」。
2. 页面主体不再重复出现大号「创建目标」标题。
3. 页面主体不再出现长页面说明副标题。
4. 页面采用右侧版式：步骤卡片 + 展开的个性化安排偏好。
5. 目标名称、截止日期、每日可用时间是最核心的三个输入项。
6. 每日可用时间使用矩形胶囊按钮，不再使用圆形小按钮。
7. 补充说明为可选项，输入框高度适中。
8. 个性化安排偏好默认展开，包含工作时段、能量状态、计划强度、仪式感元素。
9. 仪式感元素文案明确说明仅作为风格参考，不影响任务安排的科学性。
