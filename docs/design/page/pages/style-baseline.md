# 页面级布局与样式基线

## 1. Purpose

本文档定义小程序的页面级布局与样式基线，用于保证不同页面在整体结构、页面节奏、顶部区域、主体区域和基础状态表达上保持一致。
本文档只处理页面级设计规范，不定义具体组件的内部结构、字段、状态或实现方式。
组件级设计应放在后续组件 feature 或 `docs/design/components.md` 中处理。

## 2. Scope

本文档适用于：

* 四个主 Tab 页；
* 需要复用同一页面壳的二级页面；
* 页面整体布局；
* 页面顶部区域；
* 页面主体内容区域；
* 页面级状态区域；
* 页面与 TabBar 的协同关系。

本文档不处理：

* 单个组件的 props、slot、字段或内部结构；
* `TaskCard`、`TodayFocusCard`、`EnergySelector`、`EmptyState` 等组件的具体设计；
* 今日页、日历页、创建目标页、我的页的完整业务结构重排；
* 数据模型、services、storage key、planner/replanner 策略；
* AI、Deepseek、tarot 的真实业务逻辑；
* feature 的自动化验证和 passing 证据。

如果执行中发现某个组件需要重构，应记录到后续组件 feature，不要在本文档中展开。

## 3. Page-Level Design Rules

页面级设计的目标是：让用户打开不同页面时，能感受到它们属于同一套小程序，但不同页面仍然保留各自职责。

统一的是页面规则，不是页面内容。

### 3.1 视觉统一

不同页面应保持一致的：

* 页面背景；
* 页面左右内边距；
* 顶部与主体的衔接方式；
* 内容分组方式；
* 页面主操作与次级操作的层级；
* 页面状态表达方式；
* 底部安全区处理方式。

不得为了统一样式，把四个页面改成完全相同的信息结构。

### 3.2 页面节奏

页面应保持稳定的纵向节奏。

要求：

* 顶部区域不要占据过多首屏空间；
* 主体内容应尽早出现；
* 页面分组之间有清晰间隔；
* 同一分组内部元素保持紧凑；
* 页面滚动时不要出现突然断层；
* 不用无业务价值的空白、插画或长文案撑页面。

### 3.3 操作层级

页面操作应有主次关系。

要求：

* 单个页面首屏原则上只保留一个主操作；
* 次级操作不能抢主操作视觉；
* 文本型入口用于低频或辅助路径；
* 危险操作必须有明确语义，不能只靠颜色表达；
* TabBar 已有入口不要在页面主体重复做成大按钮。

### 3.4 视觉边界

页面级视觉应保持低压力、轻量、清楚。

允许：

* 低压力浅色背景；
* 清晰分组；
* 克制边框；
* 少量弱阴影；
* 统一强调色；
* 短标签和轻提示。

避免：

* 大面积渐变；
* 玻璃拟态；
* 强投影；
* 高饱和背景；
* 营销式 banner；
* 多层卡片嵌套；
* 页面顶部大块空白；
* 长篇 AI 解读占据首屏。

### 3.5 业务边界

页面级样式调整不得改变数据真相来源。

不得修改：

* models；
* services；
* schemas；
* storage key；
* planner；
* replanner；
* today-suggestion；
* ai-client；
* tarot 业务逻辑；
* TabBar 路由配置。

如果页面展示需要的数据不足：

* 优先使用现有字段降级展示；
* 在 `docs/progress.md` 记录缺口；
* 不在本轮页面级样式调整中扩展数据模型。

## 4. Page Layering

一个页面可以按以下层级理解：

```text
PageRoot
├── Top Area
├── Content Area
├── Action Area
├── State Area
└── Bottom Safe Area
```

这不是强制 DOM 结构，而是页面设计时的分层方式。实际实现可以根据页面情况调整。

## 5. Top Area

顶部区域用于页面识别和必要提示。

它应该回答：

* 当前是什么页面？
* 用户现在处在哪个主要场景？
* 是否有一句必要的短提示？

要求：

* 顶部区域保持轻量；
* 与主体内容紧凑衔接；
* 不制造大块顶部留白；
* 提示文案短，不使用长段解释。

不得：

* 在页面主体重复同名大标题；
* 在 header 下方堆叠无业务价值的胶囊标签；
* 把顶部区域做成营销 banner；
* 把 AI、塔罗、MBTI、每日关键词作为顶部主视觉。

## 6. Content Area

主体区域用于承载当前页面的核心内容。

它应该回答：

* 这个页面最重要的信息是什么？
* 用户现在最需要看见什么？
* 哪些信息只是辅助？
* 哪些入口应该降级展示？

要求：

* 主体内容应尽早出现；
* 主信息优先于解释性文案；
* 内容按清晰分组呈现；
* 主信息、辅助信息、弱入口之间有明显层级；
* 页面主体不要重复 TabBar 已有入口。

不得：

* 用长副标题解释页面；
* 用装饰或空白占据首屏；
* 使用多层卡片嵌套；
* 为了填满页面增加无意义分组。

## 7. Action Area

操作区域用于承载页面动作。

它可以出现在：

* 主体内容中的关键卡片内；
* 页面底部；
* 页面局部区域；
* 空状态或错误状态中。

要求：

* 页面主操作必须清楚；
* 次级操作应弱于主操作；
* 低频入口使用文本按钮或轻入口；
* 不同操作之间的视觉权重应符合实际优先级。

不得：

* 同一区域堆多个强主按钮；
* 把普通跳转做成强主操作；
* 把主 Tab 跳转重复做成大按钮；
* 为了视觉好看打乱操作优先级。

## 8. State Area

状态区域用于表达页面的特殊状态，包括：

* 空状态；
* 加载状态；
* 错误状态；
* 不可行状态。

状态区域不需要复杂设计，但必须可理解、可恢复。

要求：

* 空状态说明当前为什么为空，并给出下一步；
* 加载态尽量轻量，不制造焦虑；
* 错误态说明发生了什么，并给出重试、返回或重新填写等操作；
* 不可行状态应说明限制原因，并给出可执行替代方案。

不得：

* 只显示“暂无数据”；
* 只显示“出错了”；
* 使用责备式文案；
* 使用大面积插画抢占首屏；
* 输出长段 AI 解读。

## 9. Bottom Safe Area

底部安全区用于避免内容和底部导航、系统手势区域冲突。

要求：

* 主 Tab 页内容不得被 TabBar 遮挡；
* 页面底部操作不得贴近系统手势区；
* 滚动内容底部应保留必要留白；
* 不要用过大的底部空白撑页面高度。

## 10. TabBar Coordination

四个主 Tab 入口为：

* 今日任务；
* 任务日历；
* 创建目标；
* 我的。

TabBar 只承担主页面切换。

页面主体不得再放置同等级的主 Tab 跳转按钮，例如：

* 去今日任务；
* 去任务日历；
* 去创建目标；
* 去我的。

非 Tab 页面可以作为页面内操作入口，例如：

* 复盘页；
* 今日全部任务页；
* 目标管理页；
* 设置页。

## 11. Relation to Feature Verification

本文档只提供页面级设计依据，不定义 feature 的 passing 标准。

具体 feature 是否完成，应由对应的：

* `feature.md`
* `verification.md`
* `docs/progress.md`

共同记录。

如果页面级样式无法在现有结构下统一，应记录问题，而不是直接重排页面业务结构。

记录内容包括：

* 页面名称；
* 无法统一的位置；
* 具体表现；
* 是否属于页面级 feature；
* 是否属于组件级 feature；
* 本轮采用的临时降级方案。
