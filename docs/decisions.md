# decisions.md

## 记录规则

- 只记录长期影响项目执行方式的决策。
- 不记录普通文件移动、文案调整或一次性修复。
- 具体流水记录放到 `docs/log/`。
- 每条决策最多保留：决策、原因、替代方案、后续触发条件。

## D001：MVP 阶段使用本地存储

### 决策

MVP 阶段默认使用小程序本地存储。页面层不得直接读写存储，必须通过 `services/storage.ts`。

### 原因

当前目标是验证“目标 -> 计划 -> 今日任务 -> 复盘 -> 重排”闭环。本地存储能降低云开发、账号体系和权限配置成本。

### 替代方案

- 微信云数据库
- 自建后端存储

### 后续触发条件

- 需要跨设备同步
- 需要账号体系
- 需要多人协作
- 需要云端 AI 调用链路持久化

## D002：系统级确认先采用轻量方案

### 决策

系统级确认先使用 `npm run verify:system`，当前等价于 `npm run build:mp-weixin`。页面相关 feature 额外记录手动冒烟。

### 原因

项目仍在早期，页面流程尚未稳定。过早引入复杂 UI 自动化会增加维护成本。

### 替代方案

- 小程序自动化 UI 测试
- 端到端测试服务
- 微信开发者工具自动化脚本

### 后续触发条件

- F04-F06 页面流程稳定
- 手动冒烟成本明显升高
- 同一页面流程多次回归失败

## D003：Harness 主文件与流水日志分离

### 决策

`docs/progress.md` 和 `docs/decisions.md` 作为主文件。后续 day1、log1、失败详情、手动冒烟详情等流水记录写入 `docs/log/`。

### 原因

主文件需要低上下文成本。流水记录会增长，放入 `docs/log/` 可以避免主交接文件越来越长。

### 替代方案

- 将所有历史追加到 `docs/progress.md`
- 将 progress 和 decisions 都放进 `docs/log/`

### 后续触发条件

- `docs/progress.md` 超过当前状态摘要职责
- `docs/log/` 中的流水记录需要进一步分层
