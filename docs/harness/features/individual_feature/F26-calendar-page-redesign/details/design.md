# F26 日历页排版设计说明

## 1. Purpose

本文档用于说明 F26 任务日历页应如何根据参考图进行页面排版设计。

本文档只负责说明：

* 页面整体排版；
* 首屏模块顺序；
* 默认界面与选中日期后的界面差异；
* 信息层级；
* 卡片轻重关系；
* 参考图对齐方式。

本文档不负责说明：

* 日历页完整产品职责；
* 数据模型；
* services 行为；
* 图标资源清单；
* 验证命令；
* passing 门禁。

页面职责以 `docs/design/page/pages/calendar-page.md` 为准。
任务边界以 F26 `feature.md` 为准。
完成证明以 F26 `verification.md` 为准。
图标资源以 `static/icons/page/calendar/icon-spec.json` 为准。

## 2. Reference Image Usage

F26 的 UI 实现必须直接参考参考图进行设计。

参考图不是颜色参考，也不是简单的圆角、阴影、渐变参考。参考图主要用于判断：

* 页面结构；
* 模块顺序；
* 首屏信息密度；
* 卡片层级；
* 默认状态与选中日期状态的差异；
* 按钮主次关系；
* 页面是否像“目标计划板”。

不要求逐像素还原。

但如果只是修改颜色、圆角、阴影，而没有让页面结构、信息组织和首屏层级明显向参考图靠拢，不能认为符合 F26。

## 3. Two Reference States

参考图包含两个关键界面状态，F26 必须同时理解和覆盖。

### State A：默认日历概览界面

对应参考图左侧。

这是用户刚进入任务日历页时看到的默认状态。

该状态的目标是让用户快速理解：

* 当前正在推进什么目标；
* 整体进度如何；
* 未来 7 天如何安排；
* 今天或默认选中日期有什么任务；
* 是否需要调整计划。

默认界面不是任务长列表，而是目标计划板。

### State B：选中日期后的详情界面

对应参考图右侧。

这是用户点击未来 7 天计划概览中的某个具体日期后看到的界面状态。

该状态的目标是让用户从“整体计划扫描”进入“单日任务查看”。

选中日期后的页面应突出：

* 当前选中日期；
* 该日期预计用时；
* 该日期任务数量；
* 该日期任务列表；
* 针对该日期的简短计划建议；
* 调整计划入口。

选中日期后的详情界面仍然属于日历页，不应退化成今日任务页。

## 4. State A Layout：默认日历概览界面

默认界面的推荐页面顺序：

```text
顶部导航栏：任务日历

页面主体：
  当前目标计划卡
  整体进度卡
  未来 7 天计划概览
  默认选中日期任务摘要
  调整计划入口

底部 TabBar
```

默认界面首屏应尽量露出：

* 当前目标计划卡；
* 整体进度或时间压力；
* 未来 7 天计划概览的开头；
* 底部 TabBar。

如果首屏只看到一个过大的目标卡，说明信息密度不足。

如果首屏直接进入任务长列表，说明页面没有形成目标计划板层级。

## 5. State B Layout：选中日期详情界面

选中日期后的详情界面推荐页面顺序：

```text
顶部导航栏：选中日期

页面主体：
  选中日期概览卡
  该日期计划提示
  任务列表
  添加临时任务入口
  AI 计划建议
  调整计划入口

底部 TabBar
```

选中日期详情界面首屏应突出：

* 日期；
* 星期；
* 预计用时；
* 任务数量；
* 当日计划建议；
* 任务列表开头。

点击未来 7 天计划概览中的某一天后，界面应有明确状态变化。

状态变化可以体现在：

* 顶部标题或日期上下文变化；
* 当前选中日期成为页面核心信息；
* 任务列表区域明显展开；
* 默认概览中的未来 7 天区域弱化或不再作为主视觉；
* 页面从“整体计划扫描”切换为“单日任务查看”。

不要求完全复制参考图右侧布局，但必须让用户明确感知：当前正在查看某一天的任务详情。

## 6. Current Goal Card

当前目标计划卡是默认界面的第一核心卡。

它负责建立页面上下文，不只是展示目标名。

建议包含：

* 当前目标；
* 截止日期；
* 剩余天数；
* 总任务数或计划任务数；
* 预计剩余时间；
* 计划状态；
* 切换目标或管理目标入口。

目标名称不能裸露成孤立大字，应带有上下文，例如：

* 当前目标；
* 截止日期；
* 计划状态；
* 剩余时间。

当前目标计划卡应比普通任务卡更有视觉权重，但不能大到把整体进度和未来 7 天概览完全挤出首屏。

## 7. Progress And Pressure Area

整体进度和时间压力应位于当前目标计划卡之后。

该区域负责告诉用户：

* 当前目标整体进度如何；
* 已完成多少；
* 剩余多少；
* 时间是否偏紧；
* 计划状态是否正常。

要求：

* 进度指当前目标整体进度，不是单个任务进度；
* 时间压力可以用剩余天数、剩余预计时间、计划偏紧等方式表达；
* DDL 7 天以内时，时间压力视觉权重可以更高；
* DDL 超过 7 天时，不强行展开远期每日任务。

该区域应短、清楚、可扫描，不写成长篇 AI 计划分析。

## 8. Seven-Day Overview

未来 7 天计划概览是默认界面的主要扫描区域。

它应展示近 7 天安排，而不是传统大月历。

日期卡建议表达：

* 星期；
* 日期；
* 任务数量；
* 预计分钟数；
* 当前选中态；
* 基础状态：有任务、无任务、已完成、缓冲、风险。

日期卡之间应便于横向比较，让用户快速判断哪天任务重、哪天是缓冲日、哪天有风险。

不要把未来所有日期的任务一次性展开到首屏。

不要使用信息密度过高的传统月历作为首屏主体。

## 9. Selected Date Task Area

选中日期任务区域用于展示某一天的任务。

在默认界面中，它可以作为下方摘要区域出现。
在选中日期详情界面中，它成为页面主体内容。

任务区域应展示：

* 当天任务；
* 最低完成线；
* 预计时间；
* 优先级；
* 状态；
* 延期或风险提示。

任务列表可以提供弱入口，例如：

* 查看全部任务；
* 调整日期；
* 移到今天；
* 添加临时任务。

这些入口本轮可以是 UI mock 或后续 feature，不要求真实修改任务日期。

任务列表不能抢走默认界面中当前目标计划卡和未来 7 天概览的主视觉。

## 10. Secondary Sections

AI 计划建议和调整计划入口属于辅助内容。

处理规则：

* AI 计划建议只写短内容，不写成长篇 AI 分析；
* 调整计划入口可以是轻卡片或弱按钮；
* 周视图 / 月视图可以是 UI mock 或后续入口；
* 远期 Stage 属于后续扩展，不作为 F26 默认页面组件展示。

这些内容不能抢当前目标计划卡、整体进度和未来 7 天概览的主视觉。

## 11. Button Hierarchy

日历页按钮层级服务“计划管理”，不是“今日执行”。

主操作可用于：

* 调整计划；
* 生成计划，仅在有目标但无计划时出现。

次级操作可用于：

* 切换目标；
* 查看 AI 建议；
* 查看全部任务；
* 周视图 / 月视图；
* 调整日期；
* 移到今天；
* 添加临时任务。

不要把多个操作都做成同等强按钮。

`开始专注` 不是日历页主操作。如出现在任务列表中，应作为弱入口，真正执行应回到今日页。

## 12. Icon Usage

图标只作为扫描锚点，不是本文档的主体。

规则：

* 关键模块可以使用图标增强识别；
* 图标必须来自统一目录；
* 不要把图标散落到页面目录或组件目录；
* 不让 agent 自行生成最终图标资源；
* 没有正式图标时，可以预留图标位或使用临时占位，但必须记录。

图标资源、颜色、命名和导出规则以：

```text
static/icons/page/calendar/icon-spec.json
```

为准。

当前 F26 只要求 7 个 SVG 图标：`target.svg`、`calendar.svg`、`week.svg`、`adjust.svg`、`sparkle.svg`、`clock.svg`、`priority.svg`。

进度、风险、完成、缓冲日、无任务、阶段分组、任务列表容器等状态优先使用 CSS badge、dot、progress bar、chip、border 或文字表达，不单独要求 SVG。

源 SVG 目录：

```text
tools/icon-export/source/page/calendar/
```

输出 PNG 目录：

```text
static/icons/page/calendar/
```

## 13. Density Rules

允许通过以下方式提升首屏密度：

* 减少卡片外边距；
* 减少无业务价值留白；
* 压缩重复文案；
* 调整卡片内边距；
* 调整字号层级；
* 合并过长说明；
* 将弱信息降级为短句或标签。

不得通过以下方式提升首屏密度：

* 删除必要模块；
* 缩小到文字难以阅读；
* 把目标计划、进度、7 天概览混成一个不可扫描的大块；
* 把任务列表提前做成默认界面的首屏主视觉；
* 修改数据模型或 services。

## 14. Design Boundaries

F26 不要求完整实现 `calendar-page.md` 中的所有产品能力。

以下能力本轮只能作为 UI mock、弱入口或后续 feature 记录：

* 切换目标；
* 周 / 月视图；
* 调整日期；
* 移到今天；
* 真实调整计划；
* Stage 模型；
* Plan 汇总字段；
* 真实 AI 计划分析。

如果当前实现缺少相关数据：

* 优先使用现有 Task / DailyPlan 数据推导；
* 无法推导时使用 fallback；
* 在 `docs/progress.md` 记录缺口；
* 不为 UI 展示修改数据模型、services、storage key、planner 或 replanner。

## 15. Reference Alignment Checklist

实现完成后，应能回答以下问题：

* 默认界面是否像参考图左侧的目标计划板？
* 选中日期后是否像参考图右侧的日期详情界面？
* 当前目标计划卡是否是默认界面的第一核心卡？
* 整体进度或时间压力是否在首屏可见？
* 未来 7 天概览是否能快速扫描？
* 点击日期后，页面是否出现明确状态变化？
* 选中日期任务列表是否有清晰层级？
* AI 计划建议是否简短？
* 调整计划入口是否可见但不过重？
* 图标是否只作为扫描锚点，而不是装饰堆叠？

如果只是修改颜色、圆角、阴影，但没有改善默认界面、选中日期界面、目标计划卡、进度压力、7 天概览和任务列表层级，不能认为已接近参考图。
