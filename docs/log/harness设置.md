# Harness Engineering 四层笔记

## 总体理解
Harness 不是一个提示词文件，也不是单纯的 `AGENTS.md`。它是 模型权重之外的一整套工程系统：指令、工具、环境、状态管理和反馈机制。它的目标不是让模型“更聪明”，而是让 agent 在真实项目里更可靠地执行任务。课程第二讲把 harness 定义为五个子系统：Instructions、Tools、Environment、State、Feedback；这也是后面所有内容的总框架。
可以把 harness 看成 agent 的“工作场地”。模型本身像工人，harness 像施工现场：图纸是否清楚、工具是否齐全、地基是否打好、质检是否严格、交接是否干净，都会直接影响最后能不能交付。
你的四层可以总结成：

| 层级       | 核心问题               | 主要目标               |
| -------- | ------------------ | ------------------ |
| 运行前      | agent 知不知道这个项目的规则？ | 建立项目地图、知识入口和约束     |
| 运行初始化    | agent 能不能开始稳定干活？   | 搭好环境、测试、任务分解和自举契约  |
| 运行中控制与反馈 | agent 会不会跑偏、假完成？   | 控制任务边界、验证完成、收集运行信号 |
| 运行后对接    | 下一个 agent 能不能无损接手？ | 留下清洁状态、更新进度、降低熵增   |

---

# 第一层：运行前 —— 让 agent 看得懂项目

## 这一层解决什么问题？
运行前阶段解决的是 上下文供给问题。agent 不能访问你脑子里的知识，也不能天然知道团队约定、架构决策、历史坑和验证方式。第三讲强调：不在仓库里的信息，对 agent 来说基本等于不存在；所以仓库要成为项目的 System of Record，也就是唯一事实来源。
这一层的目标不是“写很多文档”，而是把关键知识放到 agent 能稳定找到的位置。

## 运行前要准备什么？
最重要的是让新 agent 会话只看仓库，就能回答五个问题：
1. 这是什么系统？
2. 代码怎么组织？
3. 怎么运行？
4. 怎么验证？
5. 现在做到哪了？
对应到文件上，大概是：

| 问题      | 推荐文件                                            |
| ------- | ----------------------------------------------- |
| 这是什么系统？ | `AGENTS.md` / `README.md`                       |
| 怎么组织？   | `ARCHITECTURE.md` / 模块文档                        |
| 怎么运行？   | `Makefile` / `package.json scripts` / `init.sh` |
| 怎么验证？   | `Makefile` / `tests/` / CI 配置                   |

|现在做到哪了？|`docs/progress.md` / `docs/harness/feature_list.json` / git 历史|
第三讲给出的核心原则是：知识要靠近代码、入口文件要标准化、文档要最小但完备，并且要跟代码一起更新。比如 API 认证规则最好放在 `src/api/ARCHITECTURE.md` 附近，而不是藏在某个全局大文档里。

## `AGENTS.md` 应该怎么理解？
AGENTS.md不应该是百科全书，而应该是 路由页。第四讲专门反对“巨型指令文件”：如果把所有规则、历史教训、部署说明、风格偏好都塞进一个 600 行文件，agent 很容易出现上下文浪费、关键信息被埋没、规则冲突和维护衰减。
比较好的做法是：

```
AGENTS.md
├── 项目一句话介绍
├── 快速开始命令
├── 必须遵守的全局硬约束
├── 验证命令
└── 专题文档路由
    ├── docs/api-patterns.md
    ├── docs/database-rules.md
    ├── docs/testing-standards.md
    └── src/module/ARCHITECTURE.md
```
你可以把 `AGENTS.md` 想象成“机场导览图”，不是“整座城市百科”。它告诉 agent 该去哪里找信息，而不是把所有信息全塞在首页。

## 运行前的最低可用模板

```
# AGENTS.md

## 项目概览
这是一个微信小程序项目，用于用户登录、资料管理和订单查询。

## 技术栈
- 小程序原生框架
- TypeScript
- ESLint
- Jest / miniprogram-simulate

## 常用命令
- 安装依赖：npm install
- 静态检查：npm run lint
- 类型检查：npm run typecheck
- 单元测试：npm run test
- 完整验证：npm run check

## 全局硬约束
- 不允许绕过统一 request 封装直接调用 wx.request
- 所有页面跳转必须使用 router 工具函数
- 涉及用户信息的接口必须检查登录态
- 新增功能必须补充测试或验证说明

## 文档路由
- 小程序页面结构：docs/pages.md
- API 调用规范：src/api/ARCHITECTURE.md
- 当前进度：docs/progress.md
- 功能清单：docs/harness/feature_list.json
```

## 这一层的验收标准
运行前准备是否合格，可以用“冷启动测试”判断：开一个全新的 agent 会话，不给任何口头解释，只让它看仓库。如果它能回答“是什么、怎么组织、怎么运行、怎么验证、做到哪了”，说明地图基本合格；如果答不上来，就说明仓库里还有知识可见性缺口。第三讲明确把这个测试作为检验仓库质量的方法。

# 第二层：运行初始化 —— 先打地基，不要一上来写功能

## 这一层解决什么问题？
初始化阶段解决的是 执行环境和项目自举问题。第六讲强调：初始化和功能实现是两种不同工作。初始化阶段的目标不是交付业务功能，而是让后续所有功能开发更可靠。
常见错误是：agent 一进项目就开始写功能，写到一半才发现依赖没装好、测试框架没配、数据库迁移跑不通、lint 规则不可用。这样会把大量上下文浪费在“修地基”上。

## 初始化阶段的核心产出
第六讲把初始化产出说得很明确：可运行环境、可验证测试框架、自举契约文档、任务分解，以及一个干净的 git checkpoint。
可以整理成五个产物：

| 产物             | 含义                          |
| -------------- | --------------------------- |
| 可运行环境          | 依赖安装好，项目能启动                 |
| 可验证测试框架        | 至少有一个示例测试能通过                |
| 自举契约文档         | 告诉后续 agent 怎么启动、怎么测试、当前状态如何 |
| 任务分解           | 把大目标拆成有验收标准的小任务             |
| Git checkpoint | 初始化完成后提交一个干净基线              |

## 自举契约是什么？
自举契约就是告诉后续 agent：“这个项目现在可以怎样被接手。”
示例：

```
# 初始化契约

## 启动命令
- 安装依赖：npm install
- 启动开发环境：npm run dev
- 静态检查：npm run lint
- 类型检查：npm run typecheck
- 运行测试：npm run test
- 完整验证：npm run check

## 当前状态
- 依赖已安装并锁定
- ESLint 已配置
- TypeScript 已配置
- Jest 测试框架已配置
- 示例测试通过

## 项目结构
- miniprogram/pages/ — 小程序页面
- miniprogram/components/ — 公共组件
- miniprogram/api/ — API 封装
- tests/ — 测试文件
- docs/ — harness 文档

## 下一步
- 按 docs/harness/feature_list.json 从 F01 开始实现
- 每次只允许一个 active 功能
```

## 初始化和运行前有什么区别？
运行前更像“地图准备”：写规则、写架构、写约束。
初始化更像“施工准备”：装依赖、跑测试、验证命令、建任务清单、提交初始状态。
举个小程序例子：
运行前你写：

```
所有请求必须通过 src/api/request.ts。页面跳转必须通过 utils/router.ts。
```
运行初始化你做：

```
npm installnpm run lintnpm run testnpm run dev
```
并确认这些命令真的能跑。

## 这一层的验收标准
第六讲给出的初始化验收标准可以转化为：

```
- [ ] npm install 从零开始成功
- [ ] npm run lint 能执行
- [ ] npm run test 至少有一个测试通过
- [ ] npm run check 能作为完整验证入口
- [ ] 新 agent 只看仓库能知道怎么跑、怎么测、下一步做什么
- [ ] docs/harness/feature_list.json 或任务分解文件存在
- [ ] 初始化结果已提交到 git
```
这一层的关键思想是：不要让 agent 边打地基边砌墙。

# 第三层：运行中控制与反馈 —— 防止跑偏、防止假完成
这一层是整套 harness 最关键、也最容易混淆的部分。它不是单纯“给 agent 一些反馈”，而是要建立一个约束系统：限制它一次做多少事，规定它怎么算完成，强制它用执行结果证明自己做对了。
它主要覆盖第七讲到第十一讲。

---

## 3.1 用 WIP=1 限制任务边界
第七讲讲的是 agent 很容易 overreach：你让它做“用户认证”，它可能同时改数据库、写前端、接邮件服务、重构错误处理、整理测试目录，最后每件事都做了一半。课程建议默认使用 WIP=1，也就是任何时刻只允许一个功能处于进行中。
WIP=1 的规则可以写成：

```
## 工作规则
- 每次只做一个功能点
- 当前功能点验证通过前，不允许开始下一个功能
- 不允许“顺便重构”无关模块
- 如果发现额外问题，记录到 backlog，不要直接展开
```
这条规则对 agent 特别重要，因为 agent 生成新想法的成本很低，很容易“顺手多做一点”。但对工程系统来说，多做一点往往意味着验证边界变复杂，最后没有一个功能真正通过。

---

## 3.2 用功能清单约束 agent 该做什么
第八讲把功能清单称为 harness 原语。它不是普通 TODO，而是调度、验证、交接和进度追踪的共同数据源。每个功能项至少应该有三元组：行为描述、验证命令、当前状态。
一个简单的 `docs/harness/feature_list.json` 可以这样写：

```
[
  {
    "id": "F01",
    "title": "用户登录",
    "behavior": "用户输入手机号和验证码后，可以成功登录并进入首页",
    "verify": "npm run test:login && npm run e2e:login",
    "status": "not_started"
  },
  {
    "id": "F02",
    "title": "密码重置",
    "behavior": "用户输入手机号后可以收到重置验证码，并成功设置新密码",
    "verify": "npm run test:reset-password && npm run e2e:reset-password",
    "status": "not_started"
  }
]
```
状态最好只有四种：

```
not_started → active → passing
              ↓
           blocked
```
关键规则是：功能从 active 变成 passing 的唯一方式，是验证命令执行成功。 不能因为 agent 说“我觉得可以了”就改状态。

---

## 3.3 三层终止校验：不让 agent 提前交卷
第九讲讲的是 agent 容易过早宣告完成。它可能只跑了单元测试，就说“功能完成了”，但真实系统里邮件没发出去、数据库迁移失败、端到端流程没跑过。课程强调，完成判定必须外部化，用基于执行的验证替代 agent 的自信判断。
三层终止校验可以这样理解：

| 层级          | 目标           | 小程序例子                       |
| ----------- | ------------ | --------------------------- |
| 第一层：静态校验    | 代码有没有明显错误    | ESLint、TypeScript、JSON 配置检查 |
| 第二层：运行时行为验证 | 局部功能能不能跑     | 单元测试、API mock、组件测试、启动检查     |
| 第三层：系统级确认   | 完整用户流程是否真的可用 | 自动化 UI 测试、端到端流程、真实/测试环境验证   |
对于小程序来说，可以对应成：

```
# 第一层：静态检查
npm run lint
npm run typecheck

# 第二层：运行时行为验证
npm run test
npm run test:api
npm run test:component

# 第三层：系统级确认
npm run e2e
# 或者半自动化：构建测试版 + 跑关键接口 + 检查日志
```
这三层的意义不同：
第一层证明“字没写错”。
第二层证明“局部逻辑能跑”。
第三层证明“用户真的能完成这个流程”。
所以单元测试通过不等于任务完成。第九讲和第十讲都强调，单元测试由于会 mock 依赖，很容易漏掉组件之间的接口不匹配、状态传播错误和环境依赖问题。

---

## 3.4 端到端验证：不仅要能跑，还要跑对
第十讲的核心是：完整流程跑通，才是真正验证。端到端测试不仅发现更多问题，还会改变 agent 写代码时的行为。因为 agent 一旦知道最后要跑完整流程，就会更注意组件交互、架构边界和错误路径。
以小程序“密码重置”为例，端到端验证不是测试某个函数：

```
resetPassword(phone)
```
而是验证完整流程：

```
打开登录页
→ 点击忘记密码
→ 输入手机号
→ 请求验证码
→ 输入验证码
→ 设置新密码
→ 返回登录页
→ 使用新密码登录成功
```
如果小程序 UI 自动化比较难，也可以先做半自动化：

```
1. 构建小程序测试版
2. 启动 mock API 服务
3. 调用关键接口验证数据流
4. 检查日志里是否出现 reset_password_success
5. 检查数据库/缓存状态是否符合预期
6. 手动补一次关键路径点击
```
第三层不一定一开始就要 100% 自动化。重要的是：不能只停留在“函数测试通过”。

---

## 3.5 可观测性：给 agent 装仪表盘
第十一讲讲的是可观测性。没有可观测性时，agent 失败之后不知道为什么失败，只能盲猜；下一个会话也不知道上一个会话到底发生了什么。课程把可观测性分成两层：运行时可观测性和过程可观测性。
运行时可观测性回答：

```
系统实际做了什么？
```
比如：

```
- 应用是否启动成功
- API 返回了什么状态码
- 页面有没有渲染成功
- 是否出现异常日志
- 数据库是否写入成功
```
过程可观测性回答：

```
为什么这个变更应该被接受？
```
比如：

```
- 当前任务是什么
- 验收标准是什么
- 哪些文件被改了
- 哪些验证通过了
- 哪些问题被阻塞了
```
对于小程序项目，可以先做一个很小的“仪表盘”：

```
# RUN_LOG.md

## 当前任务
F02 密码重置

## 修改文件
- pages/reset/reset.ts
- pages/reset/reset.wxml
- src/api/auth.ts
- tests/reset-password.test.ts

## 已执行验证
- npm run lint ✅
- npm run test:reset-password ✅
- npm run e2e:reset-password ❌

## 失败原因
e2e 流程中验证码接口返回 500。
可能原因：mock server 未配置 /api/send-code。

## 下一步
先修复 mock server，再重新执行 e2e。
```
这能让 agent 失败后更容易修正，也能让下一个会话快速接手。

# 第四层：运行后对接 —— 留下清洁状态

## 这一层解决什么问题？
运行后对接解决的是 跨会话连续性和熵增问题。第五讲说长任务很容易在会话切换时丢失上下文：新会话不知道上次做了什么、为什么这么做、测试跑到哪里。第五讲把进度日志、验证记录和下一步行动称为连续性工件。
第十二讲进一步强调：每次会话结束时必须留下 clean state。否则下一个 agent 会话会先花大量时间收拾烂摊子，比如构建失败、测试红了、临时文件没删、进度没更新。

## 什么是清洁状态？
第十二讲定义的清洁状态包括五个条件：

```
构建通过测试通过进度已记录无过时临时工件标准启动路径可用
```
也就是说，运行后不是“我代码写完了就走”，而是要像交班一样，把现场收拾到下一个人能马上继续工作的状态。

## 会话结束前应该做什么？
可以用这个退出检查表：

```
# 会话退出检查表

## 1. 验证状态
- [ ] npm run lint 通过
- [ ] npm run typecheck 通过
- [ ] npm run test 通过
- [ ] npm run check 通过或记录失败原因

## 2. 功能状态
- [ ] docs/harness/feature_list.json 已更新
- [ ] 当前任务状态正确：not_started / active / blocked / passing
- [ ] passing 状态有对应验证证据

## 3. 进度记录
- [ ] docs/progress.md 已更新
- [ ] 写明本次完成了什么
- [ ] 写明还没完成什么
- [ ] 写明下一步建议

## 4. 清理现场
- [ ] 删除临时 debug 文件
- [ ] 删除无用 console.log
- [ ] 删除注释掉的大段废代码
- [ ] 不留下无解释 TODO

## 5. 交接
- [ ] 记录关键决策
- [ ] 记录已知风险
- [ ] 记录阻塞项
- [ ] 提交 git checkpoint
```

## 一个 `docs/progress.md` 示例

```
# docs/progress.md

## 当前状态
正在实现 F02：密码重置。

## 本次完成
- 新增 pages/reset 页面
- 新增 resetPassword API 封装
- 新增验证码发送 mock 接口
- 补充 reset-password 单元测试

## 验证结果
- npm run lint ✅
- npm run typecheck ✅
- npm run test:reset-password ✅
- npm run e2e:reset-password ❌

## 未完成 / 阻塞
- e2e 流程中验证码接口偶发 500
- 需要检查 mock server 的 /api/send-code 路由

## 下一步
1. 修复 mock server 路由
2. 重跑 npm run e2e:reset-password
3. 通过后将 F02 状态改为 passing
```

## 这一层的核心原则
运行后对接不是写总结，而是降低下一个会话的恢复成本。
好的交接应该让新 agent 不用问人，就能知道：

```
上次做了什么？为什么这么做？验证到哪一步？哪里失败了？下一步应该干什么？哪些东西不能碰？
```

# 四层之间的关系
这四层不是平行的，而是一个生命周期：

```
运行前
↓
让 agent 有地图、有规则、有知识来源

运行初始化
↓
让项目能启动、能测试、能被接手

运行中控制与反馈
↓
让 agent 一次只做一个任务，并用验证证明完成

运行后对接
↓
留下清洁状态，让下一个会话无损接手
```
更短地说：

```
运行前：别让 agent 猜项目。
初始化：别让 agent 在烂地基上写功能。
运行中：别让 agent 跑偏或假完成。
运行后：别让下一个 agent 收拾烂摊子。
```

# 最小可落地版本
如果你现在想在自己的小程序项目里实践，不需要一开始做很复杂。可以先准备这 5 个文件：

```
AGENTS.md
docs/progress.md
docs/harness/feature_list.json
Makefile 或 package.json scripts
docs/testing.md
```
最小命令可以是：

```
{
  "scripts": {
    "lint": "eslint miniprogram/**/*.ts",
    "typecheck": "tsc --noEmit",
    "test": "jest",
    "check": "npm run lint && npm run typecheck && npm run test"
  }
}
```
最小规则可以是：

```
## Agent 工作规则
- 每次只做 docs/harness/feature_list.json 中的一个任务
- 当前任务验证通过前，不允许开始下一个任务
- 不能只凭代码完成度宣告完成
- 完成必须运行 npm run check
- 如果涉及用户流程，必须补充端到端验证或手动验证记录
- 会话结束前必须更新 docs/progress.md
```
这样你就已经有一个基础 harness 了。
